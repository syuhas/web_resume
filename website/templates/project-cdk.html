AWS CDK

Overview

This project involves the installation and operation of the AWS CDK. The AWS CDK is an IDE-based framework that allows us to define infrastructure as code (IaC) and uses CloudFormation to provision the resources. This is an alternative to using the CloudFormation templates themselves, as the CDK actually auto-generates the templates based on the code you define in your IDE. This is a great solution for those who prefer to use a programming language of their choice to define and provision AWS infrastructure. For the following exercise, we will define a simple VPC with 3 public facing subnets, in 3 separate availability zones.

Installing Typescript and the CDK

First, there are some prerequisties that need to be in place before actually installing the CDK. As a preliminary measure, the AWS CLI must be installed and configured with you account credentials on the system being used. Node.js is also required to use the CDK, so that must be installed beforehand. Here we will be installing Typescript and the CDK itself. In addition to this, things like Javascript and any other preferred languages that we want to use (Python, C#, etc.) must be installed. However, for our purposes here, the minimum requirements are Node.js and Typescript.

Using the Node Package Manager(npm) in global mode, we can install Typescript which will be used for this project.

Now the Cloud Developement Kit can be installed. To verify our installation, we can run a 'cdk --version' check. Additionally, a pip dependencies overview let's us see exactly what the CDK has installed on top of our previously existing dependencies.

Bootstrapping and Initialization

Once the CDK is installed, one last step is required before we can initialize our project and start coding in some infrastructure. Our account and region needs to be Bootstrapped to the CDK, which is essentially the process where containers are created behind the scenes that will contain the necessary assets and files that Cloudformation will use when we get to our deployment stage. Any additional accounts or regions we wish to use must be bootstrapped as well.

Once the bootstrapping stage is complete, we will be able to initialize a new project and start defining our infrastructure.

The bootstrap process using our account number and region. This may be done mulitple times for additional accounts and regions. Here we can see CloudFormation creating our stack preliminary resources, policies and repository that will be used with our project.

Once completed, we are now ready to begin initializing the project. For our purposes, a simple typescript blank project is created. We are going to add resources directly from our IDE which will then trigger Cloudformation to add those resources to our stack.

We can see the file structure here that the initialization process created. The key elemements here are:

cdk-demo.ts: This is where we are going to create and load all of the stacks that we define.

cdk-demo-stack.ts: This is where we actually define the stack, including reasources and properties.

package.json: Dependencies and build scripts are defined here (npm watch, npm build, npm test).

cdk.json: Finally, this instructs the toolkit how to run your application.

We can now see confirmation that our cloudformation stack has been created by signing into the console. From here we can see everything that has been created from the CDK.

In addition, we can go into the template designer to get a visual representation of what preliminary resources and roles were created in the stack.

Defining AWS Infrastructure as Code

At this point we have all of our local dependencies installed and the project stack has been initialized. Now we can begin to define our infrastructure as code using TypeScript which will later be compiled into JavaScript and verified. We will then deploy the infrastructure that we have defined in the final step. In this project, we are going to create a simple VPC with 3 public subnets.

This preliminary measure, while optional, will define the environment for our stack using the account and region that the stack will be deployed to. If we skip this step, the stack will be considered environment-agnostic which will allow it to be deployed to any environment, but certain features and context lookups will be unavailable(eg. ec2.Vpc.fromLookup).

In order to actually start defining our infrastructure, we must first install the EC2 construct library module.   A construct is essentially a dependency for the CDK that represents a single or multiple AWS resources. The EC2 construct module contains support for AWS VPCs. Using Node Package Manager(npm install @aws-cdk/aws-ec2), we can install the module that will allow us to define our VPC and subnets in the stack. We then import the module into our project

Now we essentially have a working skeleton for our stack definition. Were we to build and deploy our CDK now, no resources would be added. In the final step we will actually define the resources.

As seen here, we define a VPC with a maximum of 3 Availability Zones. Our 3 Availability Zones will each have a public subnet. In order for the subnets to be properly defined, they must be configured. The subnet mask is set to /24 which will give us 254 usable IP addresses for each subnet. Now that our VPC and subnets are defined, we can build and deploy the stack in the next step.

Building and Deploying the Stack

Finally we have laid the groundwork to deploy our stack. As we will see, there is a lot going on behind the scenes and through simple defined infrastructure you can really start to see the power of the Cloud Development Kit. In this process we are also compiling the Typescript into JavaScript and then deploying the code as JavaScript. This will also verify that the infrastructure has been properly defined and that we haven't made any mistakes along the way.

Using NPM, the code is compiled into JavaScript and checked for validity. After we confirm that no errors are present, we can use the CDK to deploy the stack. We can see here that the stack code is valid and the change to the stack is initializing and preparing to deploy.

During deployment, we can see exactly what is deploying into the stack as the resources are created.

In addition to viewing the change set progress in the IDE, we can log into the CloudFormation via the console and view the stack as the changes are in progress. We can confirm here that stack is successfully deployed.

As another visual aid, I logged into the console to check that all of the resources that I defined were properly deployed. Here we see that the VPC is available in our account...

...and the public subnets are properly deployed, one in each of our 3 Availability zones. Each has a subnet mask of /24 and the necessary route tables and internet gateways defined. The stack has now been successfully deployed and we can deploy more change sets to the stack via our code to add or remove resources as needed.

Summary

For me, this project really highlighted the power and utility of the Cloud Development Kit. It is an extremely powerful tool, being able to define and deploy entire stacks of CloudFormation-generated infrastructure using some code and your favorite IDE. As an alternative to CloudFormation templates, some may prefer this as the infrastructure can be defined using a language that some feel either more comfortable with or proficient in.

The ease at which you can provision and destroy architectures could have great applications for copying entire infrastructures over to another region for higher availability and resiliency, and in a disaster recovery plan to automate recovery to reduce RPO and RTO for less downtime.

Overall, the CDK is a great tool, a great alternative to CloudFormation templating and a very useful weapon to have in one's cloud arsenal.
